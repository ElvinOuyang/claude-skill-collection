# Claude Design Inbound

Land a Claude Design artifact into the codebase as a stack of small, reviewable PRs in the right order. The merge order is the whole point — landing tokens and implementation in the same PR has bitten this team before (Coverage Board), and the redesign work that *did* go cleanly (Mine / Tasks / Detail) used this exact shape.

## The stack shape

| PR | Contents | Why separate |
|---|---|---|
| **#1 — Tokens** | Color, spacing, typography, radius, motion variables only. No component changes, no screen changes. | Tokens are tiny + mechanical. Easy review, no behavior risk. Lands first so #2 can rebase onto a stable token surface. |
| **#2 — Implementation** | Components and screens that consume the tokens from #1. | Big PR, but every visual change is a token reference. Reviewers can focus on layout/behavior, not "is this color right?" because the token already passed review. |
| **#3 — Assets** *(optional)* | Illustrations / icons generated by `chatbot-asset-pipeline`. | Asset generation is its own lengthy process and shouldn't block #2. Lands last because #2 can ship with placeholder art. |

Stacked branching: `feature/redesign-tokens` → `feature/redesign-impl` (branched off tokens) → `feature/redesign-assets` (branched off impl). Each PR targets the previous, so review can proceed in parallel without merging blockers.

## Inputs

1. **The Claude Design artifact.** A link, an exported bundle, a Figma export, or a folder of files dropped in `design/inbound/`.
2. **The target feature scope.** Which screens / components are in this redesign? Don't try to land "all of the design system at once" — scope to what's actually changing.
3. **Trunk branch name** — usually `master` or `main`. Always fetch and rebase before branching.

## The flow

### Phase 0: Triage the artifact

Open the artifact and inventory what's in it:

- **Tokens** — color, spacing, type, radius, motion. Note anything new vs the existing token set.
- **Components** — buttons, cards, inputs, etc., with variant tables.
- **Screens** — full layouts.
- **Assets** — illustrations, icons, photography.

If the artifact is mixed (e.g., screens reference tokens that don't yet exist in the artifact), flag it to the user before splitting — adding tokens by inference is a recipe for drift.

### Phase 1: PR #1 — tokens

Branch:
```
git fetch origin && git checkout -b feature/<name>-tokens origin/<trunk>
```

Diff the artifact's tokens against the project's current token file(s). For each difference:

- **New token** — add it. Keep ordering / naming convention consistent with the existing file.
- **Changed value** — update it, and grep the codebase for raw uses of the old value so #2 catches them.
- **Removed token** — keep it if it's still referenced; mark it `// deprecated` with a TODO. Removal happens in #2 once consumers migrate.

Commit with a message that lists the deltas in plain English ("add `accent.warning`, change `spacing.md` from 12pt to 16pt, deprecate `legacy.purple`"). Push and open the PR against trunk.

### Phase 2: PR #2 — implementation

Branch from #1's branch (not trunk):
```
git checkout feature/<name>-tokens && git checkout -b feature/<name>-impl
```

Implement the components and screens, every visual value referenced as a token (`Theme.spacing.md`, not `16`). For each screen in the artifact:

1. Read the spec carefully — call out anything ambiguous before writing code; ambiguity is the #1 source of rework.
2. Build the component/screen.
3. Snapshot or screenshot it and visually compare against the artifact.
4. Note any token gaps that surface during implementation — surface them back to PR #1 rather than hard-coding values in #2.

Keep #2 reviewable. If it's growing past ~1500 LOC of changes, split by surface (e.g., `feature/<name>-impl-tasks-list`, `feature/<name>-impl-detail`) — all branched off the tokens PR.

Push and open the PR targeting the tokens branch (not trunk). Mark the description with: "Stacked on #<tokens-PR-number>, merge that first."

### Phase 3: PR #3 — assets (optional)

If the artifact includes illustrations that need generation, branch from #2's branch and hand off to `chatbot-asset-pipeline`:

```
git checkout feature/<name>-impl && git checkout -b feature/<name>-assets
```

The asset pipeline skill handles style lock + generation + catalog placement. This skill's only job here is the stacking — make sure the assets PR targets `feature/<name>-impl`, not trunk.

If there's no `design/style-lock.md` yet, hand off to `visual-ideation-moodboard` first so the asset PR has a locked style to generate against.

### Phase 4: Merge order ritual

This is the part that's gone wrong before. When PRs are approved:

1. Merge **#1 tokens** into trunk first.
2. Rebase **#2 impl** onto the merged trunk. Resolve any token-file conflicts in favor of the merged tokens (don't re-introduce changes from #1 that already landed).
3. Merge **#2 impl** into trunk.
4. Rebase **#3 assets** onto trunk and merge.

Never merge #2 before #1 — the impl will reference tokens that don't exist on trunk yet, breaking everyone else's builds.

If the user is using a stacked-PR tool (Graphite, ghstack, spr), let it handle the rebasing — but verify the order before clicking "merge stack."

## Outputs

- 2–3 PRs in stacked order, each with a clear title and description
- A `design/inbound/<name>/` directory containing the original artifact for future reference
- A short handoff note in the PR description for #2 explaining how to test the new screens

## Common failure modes

- **Single mega-PR with tokens + impl + assets.** Review takes forever, conflicts compound, and rollback becomes "revert the redesign" instead of "revert the broken token." Always stack.
- **Merging #2 before #1.** Build breaks for everyone. Enforce the merge order; don't trust intent.
- **Hard-coding values "to unblock #2".** That defeats the whole stack. Push the gap back to #1 instead.
- **Deleting deprecated tokens in #1.** Consumers haven't migrated yet — keep the deprecation in #1, do the deletion in a follow-up after #2 lands.

## Cross-skill handoff

- **Downstream:** `chatbot-asset-pipeline` (PR #3) — needs `design/style-lock.md` from `visual-ideation-moodboard`.
- **Sibling:** `design-spec-audit` — once the stack is merged, run an audit against the artifact spec to catch any implementation drift that slipped through review.
- **Reverse:** `claude-design-outbound` — for sending the current implementation *back* to Claude Design for a follow-up round.
